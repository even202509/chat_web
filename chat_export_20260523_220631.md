这个配置要求 session cookie 必须通过 HTTPS 传输。但在生产部署中，HTTPS 通常由 Nginx/反向代理终止，Flask 实际收到的是 **HTTP 请求**。由于 Flask 没有被告知"原始请求是 HTTPS"（没有 `ProxyFix`），`request.is_secure` 返回 `False`，导致：

1. **GET 请求时**：Flask 渲染 `login.html`，尝试设置 session cookie → 但 `Secure` 标志导致 Flask 认为当前连接不安全，拒绝设置或设置了浏览器无法接收的 cookie
2. **`{{ csrf_token() }}`** 生成的 CSRF token 存储在 session 中 → session 为空/无效 → CSRF token 没有可靠的 session 绑定
3. **POST 请求时**：浏览器要么没发 session cookie，要么 session 中没有有效 CSRF token → Flask-WTF 的 `CSRFProtect` 验证失败 → 返回 400（可能不是 JSON 格式，前端 `catch` 捕获显示 "Network error"）

这解释了为什么：
- GET 请求正常（不需要 CSRF，不需要 session）
- POST 请求"无响应"（CSRF 验证在进入视图函数之前就被拦截，你的业务日志代码根本没执行到）
- 本地开发正常（`localhost` 没有代理，直接 HTTP 通信，`SESSION_COOKIE_SECURE` 可能被你改过或本地没这个配置）

### 缺少的关键配置

`app.py` 中**没有**以下任一配置：

- `ProxyFix` 中间件
- `WTF_CSRF_ENABLED` 显式配置
- `app.wsgi_app = ProxyFix(app.wsgi_app, ...)`

---

## 修复方案

需要在 `app.py` 的 `csrf = CSRFProtect(app)` 之前添加 `ProxyFix` 中间件。根据你的代理层数选择：

**单层代理（最常见，Nginx → Flask）：**

```python
from werkzeug.middleware.proxy_fix import ProxyFix

# 在 SECRET_KEY 配置之后、CSRFProtect 之前添加
app.wsgi_app = ProxyFix(app.wsgi_app, x_for=1, x_proto=1, x_host=1, x_prefix=1)
```

**多层代理（CDN + Nginx → Flask）：**调整参数为代理层数。
